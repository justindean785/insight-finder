# Edge-function redeploy & deploy-drift runbook (`osint-agent`)

The recurring operational pain has been that the deployed `osint-agent` edge
function drifts from `main` and there was no way to *see* the drift. This runbook
makes drift detectable (a git-SHA build marker on `/health`) and documents the
one correct redeploy channel.

## Deploy topology (why `supabase functions deploy` is NOT used)

> A mirror merge is NOT a deploy. The only proof of deploy is a moved build SHA.

The Supabase project `skzqwbyvmwqarfgfvyky` is owned by **Lovable Cloud**, not the
user. The Lovable project is connected to its mirror repo
`justindean785/seeker-spark-search-5362c57c` (branch `main`), but **a push/merge to
that mirror only SYNCS the code into the Lovable project — it does NOT deploy the
edge function.** The deploy is a separate, explicit step (the Lovable project agent
running `supabase--deploy_edge_functions`). The user's token **403s** on
`supabase functions deploy` for this project — that is the wrong channel. CI
(`.github/workflows/ci.yml`) is **test-only by design**; it does not (and should
not) deploy.

```
edit here (insight-finder/main)
   └─ stamp build-info.ts (npm run stamp:build) → commit
        └─ surgical sync PR → seeker-spark-search-5362c57c (gh token) → merge = SYNC ONLY
             └─ EXPLICIT Lovable deploy (supabase--deploy_edge_functions) → verify health SHA moved
```

## Build marker — how drift is now detectable

`/health` returns a `build` field derived from the git short SHA, generated by
`scripts/stamp-build.mjs` into `supabase/functions/osint-agent/build-info.ts`
(a committed file — Lovable's pipeline does not run npm build steps for edge
functions, so the marker must be committed, not computed at deploy).

```bash
curl -s "https://skzqwbyvmwqarfgfvyky.supabase.co/functions/v1/osint-agent?health=1" \
  -H "Authorization: Bearer <USER_JWT>" | jq '{version, build, build_committed_at, ok, checks}'
```

Drift check: compare the live `build` SHA against recent `git log --oneline` on
`insight-finder/main`. If it is the latest deployed commit → in sync. If it is a
weeks-old SHA → the deploy has drifted; redeploy.

> One-commit offset: the marker holds HEAD *at stamp time*, so after you commit
> the stamped file the marker names that commit's parent. That is fine for drift
> detection (you are checking "is this recent?", not byte-identity).

## Redeploy procedure

```bash
# 1. From insight-finder repo root, refresh the marker to current HEAD:
npm run stamp:build            # writes supabase/functions/osint-agent/build-info.ts
git add supabase/functions/osint-agent/build-info.ts
git commit -m "chore(osint-agent): stamp build marker for redeploy"
git push

# 2. Sync the changed function into the Lovable mirror (true merge, never
#    rsync --delete). Per CLAUDE.md, sync ALL functions that changed — here just
#    osint-agent:
#      cp -R supabase/functions/osint-agent/. <mirror-clone>/supabase/functions/osint-agent/
#    then open a PR against seeker-spark-search-5362c57c, review the diff, merge.

# 3. EXPLICITLY trigger the Lovable edge-function deploy — merging the mirror PR
#    does NOT deploy. The Lovable project agent must run
#    supabase--deploy_edge_functions for osint-agent (a "redeploy" nudge often only
#    creates a commit; the merge alone only syncs code). THEN verify parity:
curl -s ".../osint-agent?health=1" -H "Authorization: Bearer <JWT>" | jq .build
#   → must equal the short SHA you shipped. If it did NOT move, nothing shipped.
```

## Orchestrator readiness gate

`/health` reports `checks.orchestrator.ok` (true when `MINIMAX_API_KEY` OR
`LOVABLE_API_KEY` is set) and `checks.core.ok` (Supabase URL + service key).
Telemetry 2026-06-27 confirms the orchestrator is live on MiniMax-M2.7
(`MINIMAX_API_KEY` set). Confirm `checks.orchestrator.ok === true` after any
secret change.

## Related drift: database migrations

Edge-function drift has a DB-migration sibling. The `save_agent_memories`
dedupe fix (`supabase/migrations/20260612_memory_save_dedup_batch.sql`, commit
`0d1827a`) is correct in source but was still erroring in production through
2026-06-27 — i.e. the migration was not applied to the live DB. Apply pending
migrations through the Lovable/Supabase channel as part of any redeploy.
