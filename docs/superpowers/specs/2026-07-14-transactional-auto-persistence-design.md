# Transactional Auto-Persistence Design

## Goal

Make structured investigation findings persist deterministically, deduplicate by
canonical identity, and produce an artifact-linked chain-of-custody row in the
same transaction. Persistence must not depend on the orchestrator choosing to
call `record_artifacts`.

## Success criteria

1. Every accepted automatically extracted finding has
   `metadata.auto_recorded = true`.
2. Every newly persisted finding has an `evidence_log` row whose `artifact_id`
   references that artifact.
3. Equivalent values collapse within a thread and kind:
   - email and domain: trimmed and lowercased;
   - username: trimmed, lowercased, and stripped of a leading `@`;
   - phone: parsed to canonical E.164; ambiguous numbers are rejected rather
     than compared by national digits;
   - domain: trailing dot removed and IDNA-normalized;
   - URL-backed findings: normalized scheme/hostname and default port while
     preserving meaningful path and query;
   - IP: canonical IPv4/IPv6 representation;
   - name: Unicode-normalized with folded whitespace but punctuation preserved;
   - address: conservatively normalized without lossy abbreviation expansion;
   - hash/wallet: canonicalized according to its network/algorithm.
4. Concurrent retries return the existing artifact and do not append duplicate
   evidence.
5. Existing evidence-chain verification remains byte-compatible.
6. Telemetry kinds are never generated by automatic finding extractors.

## Scope

This slice introduces:

- a pure artifact-candidate normalizer;
- a tool-result extractor registry for structured provider outputs;
- one edge-function recorder used by automatic extraction and all existing
  artifact-producing paths;
- a `normalized_value` artifact column and matching unique index;
- a revised transactional `record_artifacts_with_evidence` RPC;
- regression, concurrency, migration, and hash-chain compatibility tests.

This slice does not create `thread_telemetry`, redesign report rendering, change
confidence policy, or deploy production. Those remain separate changes.

## Architecture

### Artifact candidate

Extractors emit a narrow internal type:

```ts
interface ArtifactCandidate {
  kind: string;
  value: string;
  source: string;
  sourceUrl?: string | null;
  discoveredVia: string;
  rationale: string;
  confidence?: number | null;
  metadata?: Record<string, unknown>;
  autoRecorded: boolean;
}
```

Extractors cannot assign IDs, calculate custody hashes, or write to Supabase.
They only translate a successful, schema-validated tool result into candidates.
Unknown or ambiguous shapes emit no candidate and log a structured diagnostic;
they never guess.

### Extractor registry

The registry is keyed by runtime tool name and uses explicit parsers, not a
generic recursive object walker:

```ts
const extractorRegistry: Record<string, ToolResultExtractor> = {
  hunter_email_verifier: extractHunterEmail,
  indicia_person: extractIndiciaPerson,
  indicia_phone: extractIndiciaPhone,
  username_sweep: extractUsernameSweep,
  socialfetch_lookup: extractSocialProfile,
  serus_darkweb_scan: extractBreachResults,
};
```

Initial coverage targets stable high-value response contracts: email, phone,
username/social profile, breach, and broker results. Each extractor is
independently unit-tested against real response fixtures and distinguishes
positive findings, valid negatives, process telemetry, errors, and unsupported
shapes.

The shared provider executor invokes extraction only after a result is
classified as successful and non-empty. `{ found: 0 }`, empty arrays, health
responses, timeouts, skipped, failed, suppressed, cached errors, and negative
exposure results do not become finding artifacts. Explicit negative evidence
continues through its existing integrity path.

### Integrity normalization

Candidates pass through the existing confidence and status engine before
recording. The normalizer:

1. validates/coerces the strict kind;
2. computes `normalized_value`;
3. applies evidence confidence caps;
4. validates or repairs provenance without inventing a provider;
5. adds `auto_recorded`, `discovered_via`, rationale, source category, query
   types, and confidence reasons;
6. rejects empty values and telemetry kinds.

The original `value` remains suitable for display. `normalized_value` is solely
the canonical identity used for deduplication.

### Transactional recorder

One recorder accepts normalized rows and calls
`record_artifacts_with_evidence`. Automatic structured output, LLM
`record_artifacts`, anchor intake, attachment intake, dork intake, seed-derived
artifacts, report-salvage findings, and analyst-added evidence all use it. Their
only differences are `autoRecorded` and provenance metadata.

The RPC:

- authenticates thread ownership;
- locks the thread for evidence sequence allocation;
- upserts by `(thread_id, kind, normalized_value, source)`;
- inserts the artifact and linked custody row atomically;
- returns `{ artifact_id, inserted, evidence_id }`;
- on an idempotent duplicate, returns the canonical artifact ID and its existing
  linked evidence ID without appending a duplicate custody row;
- if a canonical legacy artifact lacks linked evidence, repairs that missing
  custody link inside the same transaction.

No artifact writer may call `append_evidence` with a null artifact ID after the
migration.

## Database migration

Add `artifacts.normalized_value text`, backfill it with conservative SQL
normalization, and retain the original `value`. Before adding uniqueness:

1. group duplicates by the new key;
2. select the oldest artifact as canonical;
3. repoint existing `evidence_log.artifact_id` references;
4. merge only safe metadata fields such as source arrays;
5. delete redundant artifact rows without rewriting custody hashes.

Create a unique index on:

```sql
(thread_id, kind, normalized_value, COALESCE(source, ''))
```

The RPC requires non-empty `normalized_value`. Direct legacy finding inserts are
removed rather than supported by a permanent fallback. RLS and grants remain
least-privilege and ownership-scoped.

## Minimum telemetry guard

The full `thread_telemetry` migration remains separate. Automatic extractors
must reject process-only kinds including `cluster_decision`, `triage_summary`,
`tool_failure`, `risk_assessment`, `pivot_decision`, and `run_health` unless a
future caller explicitly routes them to telemetry storage.

## Failure behavior

- Extraction failure: log tool name and shape reason; preserve the tool result.
- Validation rejection: do not persist; log the rejected kind/value reason.
- RPC failure: return a recording failure to the caller and emit a warning with
  the thread and tool, but do not abort the investigation stream.
- Custody insert failure: the transaction rolls back the artifact.
- Duplicate: return the canonical artifact ID and existing evidence ID with
  `inserted = false`.

## Tests

Development follows red-green-refactor:

1. normalization variants collapse while display values remain unchanged,
   including keeping `+1 916...` distinct from `+7 916...`;
2. generic successful tool results produce `auto_recorded` candidates;
3. skipped, failed, empty, and telemetry results produce none;
4. the LLM record path uses the transactional recorder and never sends a null
   artifact ID;
5. attachment and dork paths use the same recorder;
6. RPC output links artifact and evidence IDs;
7. concurrent equivalent inserts yield one artifact and one custody row, and
   both calls return the same artifact and evidence IDs;
8. mixed `append_evidence` and transactional writes still pass
   `verify_evidence_chain`;
9. migration is idempotent and safely consolidates legacy duplicates;
10. the same normalized email from two sources creates two observations while a
    repeat from one source reuses its observation;
11. email and phone scan fixtures prove every auto-recorded artifact has linked
    evidence and no finding evidence row has a null `artifact_id`.

## Rollout and verification

1. Run focused edge tests and SQL migration tests.
2. Run the complete edge suite and frontend tests.
3. Merge to source-of-truth `main`.
4. Apply database migrations through the authorized Lovable/Supabase channel.
5. Stamp the edge build, surgically sync, explicitly deploy, and verify the
   health build SHA moved.
6. Run one benign email or domain scan and require both queries to be non-zero:

```sql
SELECT count(*) FROM artifacts
WHERE metadata->>'auto_recorded' = 'true';

SELECT count(*) FROM evidence_log
WHERE artifact_id IS NOT NULL;
```

Also verify no duplicate normalized identities and a passing custody-chain
check for the new thread.
